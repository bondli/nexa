## Context

当前系统支持文章管理和知识库管理两个独立模块。知识库基于 Qdrant 向量存储，支持 RAG 检索。文章可以通过 `/knowledge/addToKnowledge` API 手动添加到知识库，但需要用户手动复制粘贴内容。

用户希望在文章列表页直接选择文章添加到知识库，无需切换页面。

## Goals / Non-Goals

**Goals:**
- 在文章列表页的操作菜单中提供「添加到知识库」入口
- 弹出知识库选择对话框
- 调用已有后端 API 完成向量化存储

**Non-Goals:**
- 不修改后端 API 接口
- 不修改知识库的 RAG 检索逻辑
- 不支持批量添加到知识库（本迭代）

## Decisions

### 1. UI 入口位置

**决定**：在 `Article/Actions.tsx` 的操作菜单中添加「添加到知识库」选项。

**理由**：
- Actions.tsx 已包含「删除文章」「移动分类」等操作菜单，符合现有交互模式
- 避免创建新的组件，保持改动最小化

### 2. 知识库列表获取

**决定**：复用 `Knowledge/context.tsx` 中的 `knowledgeList` 状态。

**理由**：
- 知识库列表已在 Knowledge 页面加载
- 通过 Context 获取数据，避免重复请求

**替代方案考虑**：
- 独立请求知识库列表 → 增加网络开销，不必要

### 3. 知识库选择交互

**决定**：使用 Modal + Select 组件。

**理由**：
- 与「移动分类」功能交互方式一致（同一文件内的 `showMovePanel` 模式）
- 用户认知成本低

### 4. 添加结果反馈

**决定**：成功后显示 success 消息，失败显示 error 消息。

**理由**：遵循现有 `Actions.tsx` 的错误处理模式。

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 知识库列表为空时 | 用户无法选择 | 添加提示：「请先创建知识库」 |
| 添加失败（向量生成失败） | 文章未添加 | 显示后端返回的错误信息 |

## Migration Plan

1. 部署前端更新
2. 用户即可使用新功能
3. 无数据迁移需求
