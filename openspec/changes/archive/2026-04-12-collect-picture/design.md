## Context

当前系统已有浏览器插件支持一键提取网页内容保存为笔记，以及桌面端笔记管理功能。用户新增需求：能够从网页中一键收藏图片，并在桌面应用中管理这些图片。

**现有系统组件**：
- 浏览器插件（browser-extension）：支持文字内容提取和笔记保存
- 桌面应用前端（frontend）：React + TypeScript，使用 antd6 组件库
- 桌面应用后端（server）：Node.js + Express + MySQL + Sequelize
- 分类模型（Cate）：支持笔记分类

**核心约束**：
- 图片上传使用已有接口 `/common/uploadImage`
- 需要复用现有分类体系或新建图片专用分类
- 遵循项目的分层架构规范（UI → Service → Data → AI）

## Goals / Non-Goals

**Goals:**
1. 浏览器插件支持鼠标悬停图片时显示收藏角标，点击弹出收藏弹层
2. 收藏弹层包含：分类下拉选择、描述输入、一键收藏按钮
3. 前端新增 Picture 页面，左侧分类导航，右侧图片列表
4. 图片按时间倒序展示，支持删除和查看大图
5. 后端实现图片分类和图片的 CRUD API
6. 数据库新建 Picture 表

**Non-Goals:**
- 不实现图片的编辑功能（description 可在保存时设置）
- 不实现图片的批量操作
- 不实现图片的 AI 分析或标签自动生成
- 浏览器插件的图片识别不涉及 AI 识别图片内容

## Decisions

### 1. 图片分类方案

**决定**：新建独立的图片分类表（PictureCate），与笔记分类（Cate）分开管理。

**理由**：
- 笔记分类和图片分类的业务逻辑不同，后续可能各自独立扩展
- 避免分类数量膨胀导致用户体验下降
- 保持模型职责单一

### 2. 前端页面结构

**决定**：在 `frontend/pages/Picture` 下创建独立页面，参考现有笔记页面的交互形式。

**理由**：
- 遵循项目结构规范，每个页面独立目录
- 复用已有的分类侧边栏组件模式
- 便于维护和扩展

### 3. 图片存储方案

**决定**：图片上传调用已有 `/common/uploadImage` 接口，返回的图片路径存入 Picture 表的 path 字段。

**理由**：
- 复用现有基础设施，避免重复开发
- 统一图片存储管理
- 需求明确指定使用该接口

### 4. 浏览器插件图片识别方案

**决定**：使用 CSS 选择器监听鼠标悬停事件，检测 img 元素并显示收藏角标。

**理由**：
- 实现简单，不依赖第三方库
- 能够覆盖大部分网页场景
- 用户交互直观

## Risks / Trade-offs

| 风险 | 描述 | 缓解措施 |
|------|------|----------|
| 跨域图片无法下载 | 部分网站图片有跨域限制 | 提示用户手动上传，或使用 Canvas 绘制绕过 |
| 大图片影响加载速度 | 图片列表加载大量图片会慢 | 使用缩略图或分页加载 |
| 分类数据一致性 | 图片分类删除时关联图片处理 | 级联设置 categoryId 为 null 或删除对应图片 |

## Migration Plan

1. **数据库迁移**：执行 SQL 创建 PictureCate 和 Picture 表
2. **后端部署**：部署新的 API 接口
3. **前端部署**：更新前端页面和浏览器插件
4. **验证**：测试完整流程（插件收藏 → 桌面查看）
